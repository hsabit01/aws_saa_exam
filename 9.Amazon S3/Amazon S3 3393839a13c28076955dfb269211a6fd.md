# Amazon S3

# Overview

**Amazon S3** is like a massive **digital warehouse** where you store files instead of boxes. You put data into **buckets**, and AWS handles the scale, durability, availability, and retrieval whenever you need it. It is AWS’s core **object storage** service and is commonly used for backups, static websites, media files, and data lakes.

**Key Use Case:**

- Choose **S3** when you need **highly scalable object storage** for files such as **images, videos, logs, backups, or website assets**.

**Exam Tip:**

If you see **“store files,” “static website,” “backup/archive,” or “data lake,” think S3**; if access patterns are unknown, think **S3 Intelligent-Tiering**.

**Mini contrast:**

**S3** = **object storage** for files. **EBS** = block storage for **one EC2 instance**. **EFS** = shared file storage for **multiple instances**.

# **S3 Lifecycle**

**S3 Lifecycle** is like setting automatic rules for items in a storage room: after some time, move old boxes to a cheaper shelf or throw them away. In S3, you create rules that **transition objects to lower-cost storage classes** or **expire/delete them automatically**.

**Key Use Case:**

- Choose **S3 Lifecycle** when you want to **reduce storage cost automatically over time** for aging data.

**Exam Tip:**

If you see **“old data becomes less frequently accessed”** or **“automatically move to cheaper storage,” think S3 Lifecycle**.

**Visual Logic (Mermaid):**

![S3-lifecycle.png](Amazon%20S3/S3-lifecycle.png)

# S3 Object Lock

**S3 Object Lock** is like sealing a file in a legal archive box so nobody can change or delete it until the retention period ends. It uses a **WORM model** (**write once, read many**) and supports both **retention periods** and **legal holds**.

**Key Use Case:**

- Choose **S3 Object Lock** when data must be **protected from deletion or overwrite** for **compliance, legal, or audit** needs.

**Exam Tip:**

If you see **“must not be deleted,” “immutable,” “WORM,” or “compliance retention,” think S3 Object Lock**.

**Visual Logic (Mermaid):**

![object-lock.png](Amazon%20S3/object-lock.png)

**Main difference:**

**Lifecycle** manages **cost and aging** of objects; **Object Lock** manages **immutability and protection**.

# S3 Object Replication

**S3 Replication** is like making an automatic copy of important files into another storage room, sometimes in the **same Region** and sometimes in a **different Region**. It helps with **compliance, data sovereignty, log aggregation, and disaster recovery** by copying objects between buckets.

**Key Use Case:**

- Choose **S3 Replication** when you need **automatic copies of objects across buckets/Regions/accounts**.

**Exam Tip:**

If you see **“cross-region copy,” “disaster recovery,” “data sovereignty,” or “duplicate to another bucket automatically,” think S3 Replication**.

**Visual Logic (Mermaid):**

![s3-object-replication.png](Amazon%20S3/s3-object-replication.png)

**Main difference:**

**Replication** = **ongoing automatic copy** of objects; **Batch Operations** = **run a bulk job** on a list of existing objects.

# **S3 Object Batch Operations**

**S3 Batch Operations** is like giving a warehouse worker a spreadsheet of millions of items and saying, “apply this action to all of them.” You create a **job** that performs **one operation** on a specified list of objects at very large scale, and S3 tracks progress and produces a completion report.

**Key Use Case:**

- Choose **S3 Batch Operations** when you must **perform the same action on many existing objects at scale**.

**Exam Tip:**

If you see **“millions/billions of existing objects”** plus **“copy/tag/restore/change in bulk,” think S3 Batch Operations**.

**Visual Logic (Mermaid):**

![s3-object-batch-operation.png](Amazon%20S3/s3-object-batch-operation.png)

# S3 Block Public Access

**S3 Block Public Access** is like putting a **master safety lock** on a storage room so nobody accidentally leaves the door open to the public. It provides settings that can block public access at the **account, bucket, or access point** level, and AWS notes these settings can override policies or ACLs that would otherwise allow public access.

**Key Use Case:**

- Choose **S3 Block Public Access** when you want a **default guardrail** to prevent accidental public exposure of S3 data.

**Exam Tip:**

If you see **“must prevent public access,” “accidental exposure,” or “secure all buckets by default,” think S3 Block Public Access**.

**Visual Logic (Mermaid):**

![s3-public-access.png](Amazon%20S3/s3-public-access.png)

**Main difference:**

**Block Public Access** is a **protective override/guardrail**; **Bucket Policy** is the actual **allow/deny rule set** on the bucket.

# S3 IAM Access

**IAM** is like the **security desk** for your AWS environment: it decides **who can get in** and **what they can do** once inside. AWS describes IAM as the service that controls **authentication and authorization** for AWS accounts and resources.

**Key Use Case:**

- Choose **IAM** when you need to manage **users, roles, and permissions** across AWS services.

**Exam Tip:**

If you see **“least privilege,” “permissions,” “roles for EC2/Lambda,” or “control access to AWS resources,” think IAM**.

**Visual Logic (Mermaid):**

![s3-iam-access.png](Amazon%20S3/s3-iam-access.png)

**Main difference:**

**IAM policies** are usually **identity-based** and attach to users/roles; **bucket policies** are **resource-based** and attach directly to the S3 bucket.

# S3 Bucket Policies

A **bucket policy** is like putting a sign on the storage room door saying **who is allowed in and under what conditions**. AWS defines it as a **resource-based policy** attached to an S3 bucket that grants access to the bucket and its objects.

**Key Use Case:**

- Choose **Bucket Policies** when you want to control access **directly at the S3 bucket level**, including cross-account access or condition-based access.

**Exam Tip:**

If you see **“grant another AWS account access to a bucket,” “restrict by VPC endpoint,” or “S3-specific access rules,” think Bucket Policy**.

**Visual Logic (Mermaid):**

![s3-bucket-policies.png](Amazon%20S3/s3-bucket-policies.png)

**Main difference:**

**Bucket Policy** controls access to the **bucket itself**; **IAM** controls what a **principal** can do across AWS.

# S3 Static Website

**S3 Static Website Hosting** lets you use an **S3 bucket like a simple web host** for files such as **HTML, CSS, JavaScript, and images**. Think of it like putting your website files on a **public bookshelf** so visitors can read them directly, without running a web server. To make it public, AWS requires turning off the relevant **Block Public Access** settings and adding a **bucket policy** that allows read access; S3 also gives you a **website endpoint** plus support for **index** and **error** documents.

**Key Use Case:**

- Choose **S3 Static Website Hosting** when you need a **low-cost, highly scalable host for a static site** with no server-side processing.

**Exam Tip:**

If you see **“static HTML/CSS/JS site,” “no servers,” or “website endpoint,” think S3 Static Website**; if you also need **HTTPS/custom edge delivery**, think **CloudFront in front of S3**.

**Visual Logic (Mermaid):**

![s3-static-website.png](Amazon%20S3/s3-static-website.png)

**Main difference:**

**S3 Static Website** serves **static files only**. **EC2 / Elastic Beanstalk** is for **dynamic apps** that run server-side code.

# S3 Versioning

**S3 Versioning** is like saving a new copy of a document every time someone edits it, instead of replacing the old one. In S3, each update gets a **new version ID**, so you can **recover older versions** if a file is accidentally overwritten or deleted. When someone deletes the current object, S3 usually adds a **delete marker** rather than permanently removing all versions.

**Key Use Case:**

- Choose **S3 Versioning** when you need protection against **accidental deletion or overwrite** of objects.

**Exam Tip:**

If you see **“recover deleted file,” “restore previous version,” or “accidental overwrite,” think S3 Versioning**.

**Visual Logic (Mermaid):**

![s3-versioning.png](Amazon%20S3/s3-versioning.png)

**Main difference:**

**S3 Versioning** keeps **multiple recoverable copies** of an object, while **S3 Object Lock** makes an object **immutable** for retention/compliance.

# Live Replication

**Live Replication** is like setting up an automatic copy machine that duplicates every **new or updated file** right after it is placed in the folder. In S3, it **automatically and asynchronously** replicates objects after they are written, but it **does not backfill older objects** that were already there before replication was turned on.

**Key Use Case:**

- Choose **Live Replication** when you want **ongoing automatic replication** for new uploads and updates.

**Exam Tip:**

If you see **“automatically replicate new objects going forward,” think Live Replication**; if old data must also be copied, think **On-demand Replication**.

**Visual Logic (Mermaid):**

![live-replication.png](Amazon%20S3/live-replication.png)

# On-Demand Replication

**On-demand Replication** is like giving a warehouse worker a list of **older files** and saying, “copy these now.” In S3, this is done with **S3 Batch Replication**, which replicates **existing objects** on demand, including objects that existed before replication was configured and objects that previously failed replication.

**Key Use Case:**

- Choose **On-demand Replication** when you need to **backfill or retry replication for existing objects**.

**Exam Tip:**

If you see **“replicate existing objects,” “backfill,” or “retry failed replication,” think On-demand Replication / S3 Batch Replication**.

**Visual Logic (Mermaid):**

![on-demand-replication.png](Amazon%20S3/on-demand-replication.png)

**Main difference:**

**Live Replication** handles **new and updated objects**; **On-demand Replication** handles **existing objects in bulk**.

# Cross-Region Replication (CRR)

**CRR** is like keeping a backup copy of your files in a **different country or city** in case you need distance for resilience, compliance, or lower latency for faraway users. In S3, **CRR** replicates objects between buckets in **different AWS Regions**.

**Key Use Case:**

- Choose **CRR** when you need copies in **another AWS Region** for **disaster recovery, compliance, or geographic latency** needs.

**Exam Tip:**

If you see **“different Region,” “DR,” “geographic separation,” or “compliance distance,” think CRR**.

**Visual Logic (Mermaid):**

![cross-region-replication.png](Amazon%20S3/cross-region-replication.png)

# **Same-Region Replication (SRR)**

**SRR** is like keeping a second copy of your files in another room of the **same building**. In S3, **SRR** replicates objects between buckets in the **same AWS Region**, which is useful for log aggregation, separate-account copies, or in-region compliance needs.

**Key Use Case:**

- Choose **SRR** when you need replication that **stays inside the same Region**, such as **aggregating logs** or keeping a copy in another account in-region.

**Exam Tip:**

If you see **“same Region,” “aggregate logs,” or “keep data in-country/in-region,” think SRR**.

**Visual Logic (Mermaid):**

![same-region-replication.png](Amazon%20S3/same-region-replication.png)

**Main difference:**

**CRR** = replicate to a **different Region**. **SRR** = replicate within the **same Region**. Both are forms of **Live Replication**. Also, **both source and destination buckets must have Versioning enabled** for S3 replication.
