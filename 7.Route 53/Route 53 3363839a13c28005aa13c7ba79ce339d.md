# Route 53

# Overview

**Route 53** is AWS’s **DNS service** — like the internet’s phonebook that turns a name like **example.com** into the right destination. It also helps with **domain registration** and **health checks**, so if one endpoint is unhealthy, traffic can be sent somewhere else.

**Key Use Case:**

- Choose **Route 53** when you need **DNS routing**, especially for **high availability**, **multi-Region routing**, or **custom traffic policies** like **latency-based**, **weighted**, or **failover** routing.

**Exam Tip:**

If you see **“domain name to AWS resource,” “global DNS,” “health-check-based failover,” or “latency routing,” think Route 53**.

**Visual Logic (Mermaid):**

![overview.png](Route%2053/overview.png)

**Similar-service note:**

**Route 53 vs CloudFront** → **Route 53** decides **where DNS traffic goes**; **CloudFront** caches and delivers content faster from edge locations.

# Record Types

**Route 53 records** are the actual DNS instructions inside a hosted zone — they tell the internet where to send traffic for names like **app.example.com**. Think of them like labeled signposts: some point to an **IP address** (**A/AAAA**), some point to another name (**CNAME**), and some are AWS-specific **Alias** shortcuts for AWS resources.

**Key Use Case:**

- Choose the right **record type** when mapping a domain to an **AWS resource**, especially when deciding between **Alias** and **CNAME**.

**Exam Tip:**

If you see **“root domain” + “AWS resource”**, think **Alias A/AAAA**, not **CNAME**.

**Visual Logic (Mermaid):**

![record-types.png](Route%2053/record-types.png)

**Main difference:**

**CNAME** points to **any DNS name**, but **Alias** points to **supported AWS resources** and is the usual exam answer for **apex/root domains** like **example.com**.

**Most testable records:**

- **A** → maps name to **IPv4** address.
- **AAAA** → maps name to **IPv6** address.
- **CNAME** → maps one domain name to another.
- **Alias** → Route 53 extension for **AWS resources** like **CloudFront, ALB/NLB, S3 website endpoints, API Gateway**, and more.

# Hosted Zones

A **Hosted Zone** is the container in **Route 53** that stores your DNS records for a domain like **example.com**. Think of it like a folder of directions: **Public Hosted Zone** gives directions to the **internet**, while **Private Hosted Zone** gives directions only **inside one or more VPCs**.

**Key Use Case:**

- Use **Public Hosted Zone** for **internet-facing domains**; use **Private Hosted Zone** for **internal DNS inside AWS VPCs**.

**Exam Tip:**

If you see **“internal app,” “private DNS,” “resolve only inside VPC,”** think **Private Hosted Zone**; if it must resolve on the **internet**, think **Public Hosted Zone**.

**Visual Logic (Mermaid):**

![hosted-zones.png](Route%2053/hosted-zones.png)

**Main difference:**

**Public Hosted Zone** = DNS records for the **public internet**.

**Private Hosted Zone** = DNS records usable only within **associated VPCs** (and hybrid setups via Resolver endpoints).

# Record TTL

**TTL (Time To Live)** is how long DNS resolvers keep a **record answer cached** before asking Route 53 again. Think of it like a note with an expiration time: a **low TTL** updates faster, while a **high TTL** reduces repeat lookups and is more cache-friendly.

**Key Use Case:**

- Use a **lower TTL** when records may change soon, such as **migrations, failover, or cutovers**; use a **higher TTL** for stable records.

**Exam Tip:**

If you see **“DNS change must propagate faster”**, think **lower TTL**; if you see **“stable record, fewer queries”**, think **higher TTL**.

**Visual Logic (Mermaid):**

![TTL.png](Route%2053/TTL.png)

**Main difference:**

**Low TTL** = faster DNS changes, more DNS queries.

**High TTL** = slower DNS changes, fewer DNS queries. Also, **Alias records** don’t let you set TTL directly; Route 53 uses the target AWS resource’s TTL behavior.

# CNAME vs Alias

A **CNAME** record says, “this name is really another name,” while an **Alias** record is Route 53’s AWS-friendly shortcut to supported **AWS resources** like **ALB**, **CloudFront**, or **S3 website endpoints**. Think of **CNAME** as forwarding mail to another address, and **Alias** as a building receptionist who already knows the exact AWS destination.

**Key Use Case:**

- Use **Alias** for **AWS resources**, especially at the **root domain** like **example.com**; use **CNAME** for **subdomains** pointing to another hostname.

**Exam Tip:**

If you see **“apex/root domain” + “AWS resource”**, think **Alias**, not **CNAME**.

**Visual Logic (Mermaid):**

![cname-vs-alias.png](Route%2053/cname-vs-alias.png)

**Main difference:**

**CNAME** can point to **any hostname** but **cannot** be used at the **zone apex/root domain**. **Alias** can point to **supported AWS resources**, **can** be used at the **zone apex**, and Route 53 does **not charge** for alias queries to AWS resources.

# Alias and Alias Targets Records

An **Alias record** is a Route 53 record that points your domain name to certain **AWS resources** without you manually managing their changing IP addresses. Think of it like a smart shortcut: instead of writing down the destination’s street number, you tell Route 53, “follow this AWS service directly.”

**Key Use Case:**

- Use **Alias records** when your domain should point to an **AWS-managed endpoint**, especially for the **root/apex domain** like **example.com**.

**Exam Tip:**

If you see **“root domain to ALB/CloudFront/S3/API Gateway”**, think **Alias record**.

**Visual Logic (Mermaid):**

![alias-recods-targets.png](Route%2053/alias-recods-targets.png)

**Main difference:**

**Alias** points to **supported AWS targets** and works at the **zone apex**; **CNAME** points to another hostname and **cannot** be used at the apex.

**Common Alias Record Targets:**

- **Elastic Load Balancer** (**ALB/NLB/Classic LB**)
- **CloudFront distribution**
- **Amazon S3 static website endpoint**
- **API Gateway custom/regional endpoint**
- **AWS Global Accelerator**
- **Elastic Beanstalk environment** in supported cases
- **VPC interface endpoint** and some other supported AWS service endpoints

**SAA shortcut:**

For the exam, the safest mental model is: **Alias = Route 53 + AWS resource**.

# Routing Policies

**Route 53 routing policies** decide **how DNS answers are returned** when you have one or more possible destinations. Think of them like a traffic officer: sometimes traffic goes to **one endpoint**, sometimes it is split **by weight**, **latency**, **location**, or sent to a **backup** if the primary fails.

**Key Use Case:**

- Choose a **routing policy** when you need DNS-based control over **traffic distribution, failover, geography, or performance**.

**Exam Tip:**

If you see **“backup site” → Failover**, **“send x% of traffic” → Weighted**, **“lowest delay” → Latency**, **“users by country” → Geolocation**.

**Visual Logic (Mermaid):**

![routing-policies.png](Route%2053/routing-policies.png)

**Main differences:**

- **Simple** → one resource, basic DNS answer.
- **Weighted** → split traffic by percentages.
- **Latency** → send users to the region with the **lowest latency**.
- **Failover** → **active-passive** using health checks.
- **Geolocation** → route based on **where the user is**.
- **Geoproximity** → route based on **user location relative to resources**, with optional **bias** to shift traffic regions.
- **Multi-Value Answer** → return multiple healthy records, improving availability with simple DNS-style load distribution.

**Easy exam distinction:**

**Geolocation** = “route by **user country/continent**.”

**Geoproximity** = “route to the **closest resource region**, optionally shifting with **bias**.

# Simple

**Simple routing policy** is the basic Route 53 option: it answers DNS queries with a **standard record**, usually for **one resource** like a single web server. Think of it like giving one default address when someone asks where to go—no smart traffic steering like **weighted** or **latency-based** routing.

**Key Use Case:**

- Choose **Simple routing** when you have **one main destination** and do **not** need advanced traffic control like percentage splits or region-based routing.

**Exam Tip:**

If you see **“single resource”** or **“basic DNS answer with no special routing logic,”** think **Simple routing**.

**Visual Logic (Mermaid):**

![simple-routing-policy.png](Route%2053/simple-routing-policy.png)

**Main difference:**

**Simple** = basic DNS to **one destination**.

**Weighted** = split traffic by **percentages**; **Latency** = send users to the **lowest-latency** endpoint. Also, with **Simple routing**, you can put **multiple values in one record**, and Route 53 returns them in **random order**; for **simple alias records**, you can point to only **one AWS resource** or **one record** in the same hosted zone

# Weighted Based Routing

**Weighted routing** lets Route 53 send DNS traffic to **multiple resources in proportions you choose**, like **80% to one server and 20% to another**. Think of it like a traffic splitter at a road intersection where you decide how many cars go down each road.

**Key Use Case:**

- Choose **Weighted routing** for **gradual deployments**, **A/B testing**, or sharing traffic across multiple healthy endpoints in controlled percentages.

**Exam Tip:**

If you see **“send 10% here and 90% there”** or **“gradual migration/canary rollout,”** think **Weighted routing**.

**Visual Logic (Mermaid):**

![weighted-based-routing.png](Route%2053/weighted-based-routing.png)

**Main difference:**

**Weighted** = traffic split by **percentages you define**.

**Simple** = basic DNS answer with **no controlled split**. Also, Route 53 weighted records use the **same record name and type**, and each record gets a **weight from 0 to 255**. A weight of **0** can effectively disable routing to that record unless all records are set to 0, in which case traffic is distributed equally.

# Latency Based Routing

**Latency-Based Routing** sends users to the AWS **Region** that gives them the **lowest network latency** among the latency records you created. Think of it like choosing the **closest open highway** instead of the physically closest city—the goal is **fastest response**, not just geography.

**Key Use Case:**

- Choose **Latency-Based Routing** when the same app runs in **multiple AWS Regions** and you want users sent to the **best-performing Region** automatically.

**Exam Tip:**

If you see **“multi-Region app”** + **“best performance / lowest delay for users,”** think **Latency-Based Routing**.

**Visual Logic (Mermaid):**

![weighted-routing.png](Route%2053/weighted-routing.png)

**Main difference:**

**Latency-Based Routing** = sends traffic to the **lowest-latency Region**. **Geolocation Routing** = sends traffic based on the user’s **geographic location** like country or continent. Also, if you add **health checks**, Route 53 can skip an unhealthy lowest-latency endpoint and return the next best healthy one.

# Health Check

A **Route 53 Health Check** is how AWS decides whether a destination should be treated as **healthy** before DNS sends users there. Think of it like a hotel front desk checking whether a room is available: **Endpoint** checks test the room directly, **Calculated** checks combine several room checks, and **CloudWatch** checks trust a monitoring alarm instead.

**Key Use Case:**

- Choose **Health Checks** when you need **DNS failover**, so Route 53 can stop returning unhealthy endpoints.

**Exam Tip:**

If you see **“fail over only when primary is unhealthy”**, think **Route 53 Health Check + Failover routing**. If the health decision comes from **metrics/alarm logic**, think **CloudWatch Health Check**.

**Visual Logic (Mermaid):**

![health-check.png](Route%2053/health-check.png)

**Main difference:**

**Endpoint Health Check** = Route 53 directly probes a specific endpoint over the internet. **Calculated Health Check** = Route 53 decides health based on other health checks, such as “at least 2 of 3 must be healthy.” **CloudWatch Health Check** = Route 53 uses the same underlying data stream a **CloudWatch alarm** monitors, which is useful when direct probing is not the right signal.

# Geolocation Routing

**Geolocation Routing** sends users to different endpoints based on **where the user is located**, such as a **continent**, **country**, or for the **United States**, even a **state**. Think of it like a company showing different store branches depending on the customer’s location, not based on speed but based on **geographic rules you define**.

**Key Use Case:**

- Choose **Geolocation Routing** when you must route users by **location**, such as **country-specific content**, **compliance**, or **regional landing pages**.

**Exam Tip:**

If you see **“route users in Europe to one app and users in Asia to another”**, think **Geolocation Routing**.

**Visual Logic (Mermaid):**

![geolocation-routing.png](Route%2053/geolocation-routing.png)

**Main difference:**

**Geolocation** = route by the user’s **geographic location**. **Latency-Based Routing** = route to the **lowest-latency Region**. **Geoproximity** = route based on the user’s location **relative to resource locations**, and it can shift traffic with **bias**. Also, AWS says geolocation routing can be used in both **public** and **private hosted zones**.

# Geoproximity Routing

**Geoproximity Routing** sends users to the resource that is **geographically closest to them**, based on the location of your **resources** and your users. Think of it like choosing the nearest store on a map, but with a special **bias** knob that can make one store attract **more** or **less** traffic than normal.

**Key Use Case:**

- Choose **Geoproximity Routing** when you want DNS traffic to go to the **nearest resource location** and you may need to **shift traffic geographically** using **bias**.

**Exam Tip:**

If you see **“route to nearest resource”** or **“shift more traffic to one region with bias”**, think **Geoproximity Routing**.

**Visual Logic (Mermaid):**

![geoproximity-routing.png](Route%2053/geoproximity-routing.png)

**Main difference:**

**Geoproximity** = route by **closeness to resource location** and optionally adjust with **bias**. **Geolocation** = route by the user’s **country/continent**. **Latency-Based Routing** = route by **lowest network latency**, not physical proximity. Also, Route 53 supports geoproximity records in **public and private hosted zones**.

# Geolocation vs Geoproximity

**Geolocation = “Where is the user from?”**

Route 53 matches the user to a **location bucket you define** such as **continent**, **country**, or **U.S. state**, then sends them to the record for that bucket. This is rule-based routing, so it is great for **localized content**, **regulatory requirements**, or **region-specific experiences**.

**Geoproximity = “Which resource is geographically closest to the user?”**

Route 53 looks at the **user’s location** and the **location of your resources** (AWS Region, Local Zone Group, or latitude/longitude for non-AWS resources), then routes to the **closest available resource**. You can also use **bias** to expand or shrink a resource’s traffic area.

**The mental shortcut:**

- **Geolocation** → you draw the **map boundaries**.
- **Geoproximity** → Route 53 draws the boundaries based on **resource closeness**, and you can **nudge** them with **bias**.

**Example:**

If you want **all users in Germany** to see the German site, use **Geolocation**. If you have apps in **Frankfurt**, **Virginia**, and **Singapore** and want users sent to the **nearest resource**, use **Geoproximity**.

**Big exam difference:**

If the question says **“users in Europe go here, users in Asia go there”**, that is **Geolocation**. If it says **“send users to the closest resource and optionally shift more traffic to one region”**, that is **Geoproximity**.

**One extra exam clue:**

For **Geolocation**, AWS recommends a **Default** record for users whose IP location doesn’t match a more specific rule. For **Geoproximity**, the standout feature is **bias**: **positive** bias expands a resource’s area, **negative** bias shrinks it.

# IP-based Routing

**IP-based routing** lets **Route 53** answer DNS queries based on the **client’s source IP range** that you define in a **CIDR collection**. Think of it like a VIP guest list: instead of routing by broad country rules, you route specific **IP blocks** to the endpoint you want.

**Key Use Case:**

- Choose **IP-based routing** when you know specific **customer/ISP/corporate IP ranges** and want to send them to a particular endpoint for **performance, cost, or custom overrides**.

**Exam Tip:**

If you see **“route based on known client IP ranges / CIDR blocks / ISP-specific traffic”**, think **IP-based routing**.

**Visual Logic (Mermaid):**

![ip-based-routing.png](Route%2053/ip-based-routing.png)

**Main difference:**

**Geolocation** routes by **user location** like country/continent. **IP-based routing** routes by **specific source IP ranges** you provide, so it is more precise when you know exactly which IP blocks should go where.

# Multi-Value Answer Routing

**Multi-Value Answer Routing** lets **Route 53** return **multiple healthy records** for the same DNS name instead of just one. Think of it like giving a customer several open checkout counters at once—if one counter is down, Route 53 leaves it out and returns only the healthy ones.

**Key Use Case:**

- Choose **Multi-Value Answer Routing** when you want **basic DNS load distribution + health-checked endpoints** without using a full load balancer.

**Exam Tip:**

If you see **“return multiple healthy IPs”** or **“simple DNS load balancing with health checks,”** think **Multi-Value Answer Routing**. Route 53 can return **up to 8 healthy records**, selected at random.

**Visual Logic (Mermaid):**

![multi-value-routing.png](Route%2053/multi-value-routing.png)

**Main difference:**

**Multi-Value** returns **multiple healthy answers** and filters out unhealthy ones. **Simple routing** is just a basic DNS answer and is **not** meant for health-aware traffic distribution, while **Failover routing** is more like **primary/backup** rather than returning several healthy answers at once.
