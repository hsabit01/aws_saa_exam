# Amazon S3 - Security

# **S3 Server-Side Encryption (SSE)**

Think of **S3** like a storage warehouse that automatically locks your files **after they arrive**. With **server-side encryption**, AWS handles the encryption on the S3 side, so your data is protected **at rest** without you encrypting it yourself first. Amazon S3 now automatically encrypts new uploads by default using **SSE-S3**.

**Key Use Case:**

- Choose this when you want **data at rest encrypted in S3** with minimal operational effort.

**Exam Tip:**

**If you see “encrypt objects at rest in S3,” think “Server-Side Encryption.”**

**Visual Logic (Mermaid):**

# **S3 Encryption with Amazon-Managed Key (SSE-S3)**

This is like storing your valuables in a bank locker where the **bank owns and manages the key**. It is the **simplest option**, and S3 uses its own managed keys to encrypt each object automatically. It is also the **default encryption behavior** for new S3 object uploads.

**Key Use Case:**

- Choose this when you want **simple default encryption** with **lowest management overhead**.

**Exam Tip:**

**If you see “basic encryption with no extra key management,” think “SSE-S3.”**

**Main Difference:**

**SSE-S3 vs SSE-KMS:** **SSE-S3** is simpler; **SSE-KMS** gives you **more control, auditing, and key policies**.

**Visual Logic (Mermaid):**

![amazon-managed-key.png](Amazon%20S3%20-%20Security/amazon-managed-key.png)

# **S3 Encryption with AWS KMS (SSE-KMS)**

This is like using a locker, but now you have a **security desk that tracks who used which key and when**. **AWS KMS** gives you stronger control over keys, access policies, and audit visibility through services like **CloudTrail**.

**Key Use Case:**

- Choose this when you need **key control, audit trails, compliance, or stricter access control**.

**Exam Tip:**

**If you see “audit,” “compliance,” “customer-managed keys,” or “control access to encryption keys,” think “SSE-KMS.”**

**Main Difference:**

**SSE-KMS vs SSE-S3:** **SSE-KMS** adds **fine-grained key control and auditing**, but can add **KMS cost and permissions complexity**.

**Visual Logic (Mermaid):**

![amazon-key-amangement-service.png](Amazon%20S3%20-%20Security/amazon-key-amangement-service.png)

# **S3 Encryption with Customer-Provided Key (SSE-C)**

This is like renting a storage unit where **you bring your own lock and key**, and the storage company never keeps the key. S3 encrypts the object for you, but **you must provide the key on every upload and download request**, and if you lose it, you lose access to the data.

**Key Use Case:**

- Choose this only when the customer must **fully provide and control the encryption key outside AWS**.

**Exam Tip:**

**If you see “customer supplies the key” or “AWS must not store the key,” think “SSE-C.”**

**Main Difference:**

**SSE-C vs SSE-KMS:** With **SSE-C**, **you** provide the key every time; with **SSE-KMS**, AWS KMS stores and manages the key lifecycle for you. SSE-C also requires **HTTPS** and is less flexible with AWS services.

**Visual Logic (Mermaid):**

![customer-provided-key.png](Amazon%20S3%20-%20Security/customer-provided-key.png)

### **Quick Exam Memory**

- **SSE-S3** = easiest, default, S3 manages keys.
- **SSE-KMS** = more control, audit, compliance.
- **SSE-C** = customer brings and manages the key; AWS does not store it.

# **S3 Client-Side Encryption**

Think of this like putting your file in a **locked box before handing it to the storage company**. With **client-side encryption**, your app encrypts the data **before** it is sent to **Amazon S3**, so S3 just stores an already-encrypted object and does not decrypt it for you.

**Key Use Case:**

- Choose this when you need **end-to-end control** and want data encrypted **before it ever reaches AWS**.

**Exam Tip:**

**If you see “encrypt before upload” or “AWS must never see plaintext,” think “client-side encryption.”**

**Main Difference:**

**Client-side encryption vs SSE-KMS:** with **client-side encryption**, **your application encrypts first**; with **SSE-KMS**, **S3 encrypts after upload** using KMS-managed keys.

**Visual Logic (Mermaid):**

![client-encryption.png](Amazon%20S3%20-%20Security/client-encryption.png)

**Memory Hook:**

**Client-side = encrypt first, then upload.**

**Server-side = upload first, then AWS encrypts it.**

# **S3 In-Transit Encryption**

**Simple Explanation:**

Think of this like sending a **sealed package through the mail** instead of a transparent bag. **In-transit encryption** protects data **while it is moving** between a client, application, and AWS service, usually with **HTTPS/TLS**, so others cannot easily read or tamper with it on the way.

**Key Use Case:**

- Choose this when data is traveling over a network and must be protected from **interception or tampering**.

**Exam Tip:**

**If you see “data moving over the network,” “HTTPS,” or “TLS,” think “in-transit encryption.”**

**Main Difference:**

**In-transit encryption** protects data **while moving**; **at-rest encryption** protects data **while stored**. For S3, AWS recommends enforcing **HTTPS-only** access and you can require minimum TLS versions in policy.

**Visual Logic (Mermaid):**

![in-transit-encryption.png](Amazon%20S3%20-%20Security/in-transit-encryption.png)

**Memory Hook:**

**At rest = stored on disk.**

**In transit = moving across the network.**

# **S3 CORS**

**Simple Explanation:**

Think of **CORS** like a **guest list for your S3 bucket**. If a website running in one domain tries to access files in your S3 bucket from another domain, **CORS tells the browser whether that cross-origin request is allowed**.

**Key Use Case:**

- Choose this when a **browser-based frontend** hosted on one origin needs to access **S3 objects** from a different origin.

**Exam Tip:**

**If you see “browser blocked request from another domain,” think “S3 CORS configuration.”**

**Main Difference:**

**CORS is not a permission system by itself** — it controls what the **browser** is allowed to do, while **bucket policies / IAM / ACLs** still control actual access.

**Visual Logic (Mermaid):**

![s3-cors.png](Amazon%20S3%20-%20Security/s3-cors.png)

**Quick Memory Hook:**

**CORS = browser cross-origin rules.**

**Bucket policy = actual S3 permission.**

# **S3 MFA Delete**

**Simple Explanation:**

Think of **MFA Delete** like putting a **two-step approval lock** on important files in S3. Even if someone has access, they still need an **MFA code** to permanently delete object versions or change the bucket’s **versioning** settings.

**Key Use Case:**

- Choose this when you need extra protection against **accidental or malicious permanent deletion** in a **versioning-enabled** S3 bucket.

**Exam Tip:**

**If you see “protect against permanent deletes in versioned S3,” think “MFA Delete.”**

**Main Difference:**

**Versioning** helps you recover deleted objects with old versions; **MFA Delete** adds an extra approval step before someone can **permanently remove versions** or change versioning. Also, only the **root user** can enable or disable MFA Delete.

**Visual Logic (Mermaid):**

![s3-mfa-delete.png](Amazon%20S3%20-%20Security/s3-mfa-delete.png)

**Quick Memory Hook:**

**MFA Delete = Versioning + MFA for permanent delete protection.**

It cannot be used with **S3 Lifecycle** and is not supported for **directory buckets**.

# S3 Access Logs

Think of **S3 Access Logs** like a **visitor logbook** for your storage bucket. They record details about requests made to the bucket, such as **who accessed it, when, and what action was attempted**, which helps with auditing and troubleshooting.

**Key Use Case:**

- Choose this when you need **request-level visibility** for an S3 bucket for **audit, security review, or usage analysis**.

**Exam Tip:**

**If you see “track bucket requests for audit,” think “S3 Server Access Logging.”** For easier event analysis, AWS generally recommends **CloudTrail data events** over S3 access logs.

**Main Difference:**

**S3 Access Logs vs CloudTrail:** **S3 Access Logs** are simple bucket request logs delivered to S3, while **CloudTrail** is usually the better exam answer when the question stresses **API auditing, easier searching, richer event details, or centralized governance**.

**Visual Logic (Mermaid):**

![s3-access-logs.png](Amazon%20S3%20-%20Security/s3-access-logs.png)

**Quick Memory Hook:**

- **Same account, same Region** for source and destination bucket.
- Best practice: send logs to a **different bucket**, because logging to the same bucket can create a **log-on-log loop**.
- Destination bucket must not use **Requester Pays** and must not have **Object Lock default retention** enabled.

# S3 Pre-signed URLs

Think of a **pre-signed URL** like a **temporary guest pass** to one specific S3 object. You generate a special link using your AWS permissions, and someone else can use that link to **upload or download** the object for a limited time without needing their own AWS credentials.

**Key Use Case:**

- Choose this when you want to give **temporary, limited access** to a private S3 object or upload path.

**Exam Tip:**

**If you see “temporary access to a private S3 object without making the bucket public,” think “pre-signed URL.”**

**Main Difference:**

**Pre-signed URL vs public object:** a **pre-signed URL** is **time-limited and private by default**; a **public object** is open to anyone until you change permissions. Also, console-generated presigned URLs can last **1 minute to 12 hours**, while CLI/SDK-generated ones can last up to **7 days**.

**Visual Logic (Mermaid):**

![s3-pre-signed-urls.png](Amazon%20S3%20-%20Security/s3-pre-signed-urls.png)

**Quick Memory Hook:**

**Backend signs it, user uses it, S3 trusts it temporarily.** If the URL was created with temporary credentials, it expires when those credentials expire, even if the URL’s configured expiration is later.

# Glacier Vault Lock

Think of **Vault Lock** like putting legal-grade **tamper-proof rules** on an archive vault. Once locked, the vault policy can enforce **WORM-style protection** so archived data can’t be deleted early, which is useful for compliance and long-term retention.

**Key Use Case:**

- Choose this for **compliance archives** where retained data must be protected from deletion or policy changes.

**Exam Tip:**

**If you see “compliance archive” + “cannot be deleted before retention ends,” think “Glacier Vault Lock.”**

**Main Difference:**

**Glacier Vault Lock** applies to **legacy Amazon S3 Glacier vaults**, while **S3 Object Lock** is the modern S3 feature for locking **S3 object versions** in buckets with WORM protection.

**Visual Logic (Mermaid):**

![glacier-vault-lock.png](Amazon%20S3%20-%20Security/glacier-vault-lock.png)

**Quick Memory Hook:**

**Vault Lock = lock the archive policy. Object Lock = lock S3 objects.** Also, after Vault Lock reaches the **Locked** state, you can’t delete that lock policy.

# S3 Object Lock

Think of **S3 Object Lock** like putting a file in a **tamper-proof records box**. Once locked, an object version cannot be **deleted or overwritten** for a set time, or until a **legal hold** is removed, which gives you **WORM** protection.

**Key Use Case:**

- Choose this when you need **compliance retention** or strong protection against **accidental/malicious deletion** of S3 data.

**Exam Tip:**

**If you see “WORM,” “regulatory retention,” or “cannot be deleted before retention ends,” think “S3 Object Lock.”**

**Main Difference:**

**Object Lock vs MFA Delete:** **Object Lock** protects specific **object versions** with retention periods or legal holds; **MFA Delete** adds MFA before permanent deletes/versioning changes on the bucket. Also, **Object Lock requires versioning**, and once enabled on a bucket, **you can’t disable Object Lock or suspend versioning** for that bucket.

**Visual Logic (Mermaid):**

![s3-object-lock.png](Amazon%20S3%20-%20Security/s3-object-lock.png)

**Quick Memory Hook:**

**Retention period = time-based lock.**

**Legal hold = stays locked until someone authorized removes it.** Governance mode can be bypassed by authorized users with the right permission, but legal holds are independent.

# **S3 Access Points**

Think of **S3 Access Points** like creating **separate doors** into the same storage building. Each door can have its own **permissions and network rules**, so different apps or teams can access the same bucket in different controlled ways.

**Key Use Case:**

- Choose this when **multiple apps, teams, or accounts** need different ways to access the **same S3 bucket**.

**Exam Tip:**

**If you see “one bucket, many access patterns/policies,” think “S3 Access Points.”**

**Main Difference:**

**Bucket policy** = one main rule set for the bucket.

**Access Points** = multiple named endpoints, each with its **own policy**.

**Visual Logic (Mermaid):**

![access-points.png](Amazon%20S3%20-%20Security/access-points.png)

# **S3 Access Points – VPC Origin**

This is like having a special door to S3 that only works **from inside your private office building**. A **VPC-origin access point** only accepts requests that come from the specified **VPC**, so internet-based requests are rejected.

**Key Use Case:**

- Choose this when S3 data must be accessed **only from private resources inside a VPC**.

**Exam Tip:**

**If you see “private S3 access only from VPC,” think “Access Point with VPC origin.”**

**Main Difference:**

**Regular Access Point** = can be internet-accessible depending on policy.

**VPC-origin Access Point** = request must come from the **configured VPC**.

**Visual Logic (Mermaid):**

![access-points-vpc.png](Amazon%20S3%20-%20Security/access-points-vpc.png)

# **S3 Access Points – Lambda Functions (S3 Object Lambda)**

**Simple Explanation:**

Think of this like putting a **smart filter** in front of S3. When someone requests an object, **Lambda** can automatically modify the response, such as redacting data, resizing an image, or changing file format, before it reaches the client.

**Key Use Case:**

- Choose this when you want to **transform S3 data on the fly** without storing multiple copies.

**Exam Tip:**

**If you see “modify S3 object during GET without changing original object,” think “S3 Object Lambda.”**

**Main Difference:**

**Access Point** = controls **how** you access the bucket.

**S3 Object Lambda** = controls **what data is returned** by adding a **Lambda transformation**. Also, as of **November 7, 2025**, S3 Object Lambda is available only to **existing customers** already using it and select APN partners.

**Visual Logic (Mermaid):**

![access-points-lambda.png](Amazon%20S3%20-%20Security/access-points-lambda.png)

**Quick Memory Hook:**

**Access Point = separate entry door.**

**VPC Origin = private door only from VPC.**

**Object Lambda = smart door that changes the object before handing it back.**
