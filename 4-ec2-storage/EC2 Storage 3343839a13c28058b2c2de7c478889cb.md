# EC2 Storage

# Elastic Storage Block (ESB)

**Amazon EBS** is like a **virtual hard drive** you attach to an **EC2** server. It keeps your data even if the instance stops, and you can take **snapshots** as backups and restore from them later.

**Key Use Case:**

- Choose **EBS** when an **EC2 instance** needs **persistent block storage**, such as for **databases, OS disks, or application files**.

**Exam Tip:**

If you see **“attach storage to one EC2 instance”**, **“low-latency block storage”**, or **“boot volume”**, think **EBS**.

**Main Difference:**

**EBS vs EFS** → **EBS** is usually for **one EC2 instance** as a block disk; **EFS** is a **shared file system** for multiple instances. This matters a lot in exam questions.

# EBS Snapshot

**EBS Snapshots** are like taking a **photo of your EBS volume** at a specific moment so you can restore it later if something goes wrong. They’re **incremental backups**, so after the first one, AWS saves only the changed blocks, which helps reduce storage use.

**Key Use Case:**

- Choose **EBS Snapshots** when you need **backup, disaster recovery, or to recreate an EBS volume** from a known point in time.

**Exam Tip:**

If you see **“point-in-time backup,” “restore volume,”** or **“incremental backup for EC2 storage,”** think **EBS Snapshot**.

**Main Difference:**

**EBS Snapshot vs AMI** → **Snapshot** backs up a **volume**; **AMI** is used to launch an **EC2 instance** and includes the needed backing snapshot(s).

![ebs-snapshot.png](EC2%20Storage/ebs-snapshot.png)

# EBS Snapshot Features

**EBS Snapshot features** are the extra powers around snapshot backups: you can make **incremental backups**, **restore new volumes**, **copy snapshots** to other Regions/accounts, and use options like **Fast Snapshot Restore**, **Archive tier**, and **Recycle Bin** for speed, cost savings, and recovery. Think of it like a backup system with **restore, move, speed-up, and safety-net** buttons built in.

**Key Use Case:**

- Choose these features when you need **better backup strategy for EBS**: faster restores, cheaper long-term retention, cross-Region DR, or protection from accidental deletion.

**Exam Tip:**

If you see **“incremental backup”**, **“restore volume quickly”**, **“long-term snapshot retention”**, or **“accidental snapshot deletion”**, think **EBS Snapshot features** like **FSR, Archive, and Recycle Bin**.

**Main Difference:**

**Fast Snapshot Restore vs Archive** → **FSR** is for **speed** when creating volumes from snapshots; **Archive** is for **low-cost long-term storage** of rarely used snapshots.

![ebs-snapshot-features.png](EC2%20Storage/ebs-snapshot-features.png)

# Amazon Machine Image (AMI)

An **Amazon Machine Image (AMI)** is like a **master template** for a server. It contains the **OS**, basic software, and settings so you can launch many **EC2 instances** with the same setup every time.

**Key Use Case:**

- Choose **AMI** when you want **repeatable EC2 deployments** or a **golden image** for launching identical servers.

**Exam Tip:**

If you see **“preconfigured EC2 template,” “golden image,”** or **“launch multiple identical instances,”** think **AMI**.

**Main Difference:**

**AMI vs Snapshot** → **AMI** launches a **full EC2 instance template**; **EBS Snapshot** backs up an **EBS volume**.

![ami.png](EC2%20Storage/ami.png)

# EBS Volume Types

**EBS volume types** are different kinds of cloud hard drives for **EC2**, each tuned for either **speed** or **cost**. Think of them like choosing between a **sports car** for performance and a **cargo truck** for moving lots of data cheaply.

**Key Use Case:**

- Choose **gp3** for most workloads, **io2** for **high-performance databases**, and **st1/sc1** for **large sequential throughput-heavy data**.

**Exam Tip:**

If you see **“general purpose SSD”**, think **gp3**; if you see **“highest IOPS / critical database”**, think **io2**; if you see **“big data / log processing / sequential throughput”**, think **st1**.

**Main Difference:**

**gp3 vs io2** → **gp3** is the default balanced choice for most apps; **io2** is for **mission-critical, low-latency, very high IOPS** workloads.

**SSD vs HDD** → **SSD** (`gp2`, `gp3`, `io1`, `io2`) is for **transactional/random I/O**; **HDD** (`st1`, `sc1`) is for **large sequential I/O** and cannot be used as boot volumes.

![ebs-volume-types.png](EC2%20Storage/ebs-volume-types.png)

# EBS Multi-Attach

**EBS Multi-Attach** lets you connect **one EBS volume** to **multiple EC2 instances at the same time**, like one shared external drive plugged into several servers. It is mainly for **clustered applications** that know how to safely manage shared reads and writes.

**Key Use Case:**

- Choose **EBS Multi-Attach** when you need **shared block storage** for multiple **EC2 instances in the same Availability Zone**, especially for **high-availability clustered apps**.

**Exam Tip:**

If you see **“shared EBS volume across EC2s,” “same AZ,”** or **“clustered application”**, think **EBS Multi-Attach** on **io1/io2**.

**Main Difference:**

**Multi-Attach vs EFS** → **Multi-Attach** is **shared block storage** for specific EC2 clustering use cases; **EFS** is a **shared file system** and is usually the easier answer when many instances need shared files. Also, **Multi-Attach** supports **io1/io2** volumes and works only with instances in the **same Availability Zone**.

![ebs-multi-attach.png](EC2%20Storage/ebs-multi-attach.png)

# Elastic File System (EFS)

**Amazon EFS** is like a **shared network drive** that many servers can use at the same time. Unlike **EBS**, which is usually attached to one EC2 instance, **EFS** lets multiple compute resources share the same files and it grows automatically as you add data.

**Key Use Case:**

- Choose **EFS** when you need **shared file storage** for multiple **EC2 instances** or containers across **multiple Availability Zones** in the same Region.

**Exam Tip:**

If you see **“shared file system,” “Linux mount,”** or **“multiple EC2 instances need the same files,”** think **EFS**.

**Main Difference:**

**EFS vs EBS** → **EFS** is **shared file storage** over **NFS** for many instances; **EBS** is **block storage** usually attached to a single EC2 instance. Also, **EFS** is for **Linux-based** workloads, not Windows EC2 instances.

![elastic-file-system.png](EC2%20Storage/elastic-file-system.png)

# EFS Use Cases and Access

**EFS access** works by creating **mount targets** inside your **VPC**, then your **Linux EC2, ECS, EKS, or Lambda** mounts the file system over **NFS** using its DNS name. Think of it like a **shared office file server** that many systems can open at once, with **access points** acting like separate doors for different apps or teams.

**Key Use Case:**

- Choose **EFS** when multiple **Linux-based** compute resources need **shared, scalable file storage** across AZs, such as **web fleets, CMS/content sharing, home directories, and container/Kubernetes shared storage**.

**Exam Tip:**

If you see **“shared Linux files,” “mount with NFS,” “multiple EC2s/containers need the same files,”** or **“app-specific access control,”** think **EFS** with **mount targets** and possibly **access points**.

**Main Difference:**

**EFS vs S3** → **EFS** is a **mounted file system** for live shared access; **S3** is **object storage** accessed with APIs.

**EFS vs EBS** → **EFS** is **shared file storage** for many clients; **EBS** is **block storage** typically for one EC2 instance. Also, **EFS isn’t supported for Microsoft Windows EC2 instances**.

![elastic-file-system.png](EC2%20Storage/elastic-file-system%201.png)

# EFS Performance and Throughput Modes

**EFS performance mode** decides the **latency style** of the file system, while **throughput mode** decides how much data it can move. Think of it like choosing whether you want a **quick-response road** or a **high-capacity highway**, then deciding whether traffic speed should be **automatic**, **reserved**, or **credit-based**. AWS recommends **General Purpose** for most file systems, and **Elastic throughput** for most workloads.

**Key Use Case:**

- Choose **General Purpose + Elastic throughput** for most latency-sensitive shared file workloads; use **Provisioned** when you need a predictable throughput level, and know that **Max I/O** is for highly parallel workloads that can tolerate higher latency.

**Exam Tip:**

If you see **“low latency shared file system”**, think **General Purpose**; if you see **“throughput scales automatically with workload”**, think **Elastic throughput**.

**Main Difference:**

**General Purpose vs Max I/O** → **General Purpose** is the default and lowest-latency choice; **Max I/O** supports highly parallel workloads but with higher per-operation latency, and it is **not supported** with **Elastic throughput** or **One Zone** file systems.

**Elastic vs Provisioned vs Bursting** → **Elastic** scales throughput automatically, **Provisioned** reserves a set throughput level, and **Bursting** uses a credit-based model tied to stored data.

# EFS Storage Classes and Lifecycle Tiers

**EFS storage classes and lifecycle tiers** let you keep active files in a **fast tier** and automatically move older, rarely used files into **cheaper tiers**. Think of it like keeping current paperwork on your desk and moving old folders to a lower-cost archive room.

**Key Use Case:**

- Choose **EFS lifecycle management** when you need **shared file storage** but want to **automatically reduce cost** for files that are rarely accessed.

**Exam Tip:**

If you see **“files become cold over time,” “shared file system,”** or **“move infrequently accessed files automatically,”** think **EFS IA / Archive with lifecycle policies**.

**Main Difference:**

**EFS Standard** is for **frequently accessed** files with the lowest latency; **EFS IA** is for files accessed only a few times per quarter; **EFS Archive** is for files accessed only a few times per year or less. Lifecycle policies can move files into **IA** or **Archive**, and optionally move them back to **Standard on first access**. Also, **Regional** and **One Zone** file systems have matching storage-class variants, like **One Zone-IA** for lower cost but lower resilience.

![efs-storage-class-lifecycle.png](EC2%20Storage/efs-storage-class-lifecycle.png)

# EBS vs EFS vs Instance Store

Think of **EBS** as a **hard drive for one server**, **EFS** as a **shared network drive for many servers**, and **Instance Store** as a **temporary local disk** attached to the machine itself. **EBS** is persistent block storage for EC2, **EFS** is shared file storage mounted over **NFS**, and **Instance Store** is temporary block storage physically attached to the host and ideal for caches or scratch data.

**Key Use Case:**

- Choose **EBS** for **persistent storage for one EC2** (boot volume, database disk), **EFS** for **shared files across multiple Linux instances/AZs**, and **Instance Store** for **temporary ultra-fast local storage** like cache, buffers, or scratch space.

**Exam Tip:**

If you see **“one EC2 + persistent block”**, think **EBS**; if you see **“many EC2s need the same files”**, think **EFS**; if you see **“temporary high-speed storage lost on stop/terminate”**, think **Instance Store**.

**Main Difference:**

**EBS vs EFS vs Instance Store** → **EBS** = **persistent block storage** attached to EC2; **EFS** = **shared elastic file system** with concurrent access from multiple EC2 instances; **Instance Store** = **temporary block storage** tied to the life of the instance and not preserved when the instance stops, terminates, or the underlying disk fails.
